# oz Plan

协助把模糊想法收敛成一个可以创建提案的边界。

## 入口分类

规划阶段必须先在 micro、small、standard 三种入口中选型。

| 类型 | 适用场景 | 产物 |
| --- | --- | --- |
| micro | 不改变用户可感知行为、命令契约、状态语义或长期规格的纯实现修复 | TDD + git commit，不创建 change 目录 |
| small | 单一业务意图，最多 2 个验收场景或 2 个 required tests，且没有复杂设计分歧 | `brief.md`、`acceptance.json`、`tests/` |
| standard | 跨模块、高风险、多场景，或超过 small 上限 | 完整提案六件套 + `acceptance.json` + `tests/` |

standard 升级触发器是风险、跨度和场景复杂度；不得为了进入 standard 硬凑测试或任务。small 仍要说明长期规格去向，归档时合并到 `docs/specs/` 和 `tests/specs/`。

## 流程

1. 先确认用户要解决的问题和当前痛点，不急着写文件
2. 拆出本次会改变的行为、文档、命令、测试或数据
3. 明确非目标，避免执行阶段把邻近问题一起做掉
4. 强调测试集的重要性，验收依靠测试，测试必须准确反映用户的真实意图；测试数量只是分类信号，不能硬凑
5. 给出提案名称让用户审核

围绕用户意图讨论：

- 本质诉求是什么，是否真的必须做，很多需求都是伪需求，一开始就应该否决，而不是浪费精力；不要迁就用户不切实际的幻想或者错误思路，该制止时要严格制止，做一个无情的幻想破碎者
- 解决哪些问题
- 哪些行为、文档、命令或测试会变化
- 哪些内容明确不在本次范围内
- 需要新增或更新哪些真实测试代码
- 是否会影响历史测试，是否需要按新意图更新旧测试
- 多用ascii图表和树状图阐述说明

## 退出条件

规划结束时必须给出：

- 提案名称：需求描述，可混用英文、数字和连字符，但不能全英文
- 问题
- 范围
- 非目标
- 测试：执行阶段应写入 `docs/changes/<提案名称>/tests/` 的真实测试代码
- 待定问题：无法合理假设的阻塞问题，如果存在会改变范围、验收标准、数据来源或权限边界的问题，先停下问用户，不要进入创建阶段。

## 反偷懒检查

| 常见偷懒理由 | 处理方式 |
| --- | --- |
| “需求很简单，可以直接改” | 仍要给出 scope、non-goals 和 tests；简单需求更需要防止越界 |
| “测试策略执行阶段再想” | 规划阶段至少要说清真实入口、断言对象和可能需要的 evidence |
| “change-name 先随便写” | 名称会成为目录和审计入口，必须用中文表达业务意图 |
