# 设计：彻底合并 wo 为 oz flow

## 命令入口

保留 `cmd/oz/main.go` 作为唯一二进制入口。`internal/ozcli` 增加 `flow` 命令组分发，把 `oz flow <subcommand>` 交给现有工作流应用层执行。

执行阶段可以继续复用 `internal/app` 的工作流实现，但需要把包注释、帮助文案、错误提示和用户可见命令名改为 `oz flow`。不接受只在 `ozcli` 中添加别名而保留 `cmd/wo` 的做法。

## 命名清理

本次变更是破坏性清理，因此运行态和配置命名也应同步收敛：

- 新配置文件使用 `oz-flow.yaml`。
- 新运行态目录使用 `$XDG_STATE_HOME/oz/flow/...` 或等价的 `oz` 命名空间。
- 模板文件和 embed 包说明不再使用 `wo-*` 文件名或 `wo` 产品描述。
- README、活跃 specs、长期 tests 和 CI 文档统一使用 `oz flow`。

历史归档提案可以作为历史记录保留旧文字；它们不再代表当前产品面。合同测试只扫描活跃产品路径，避免要求重写归档史料。

## 测试策略

创建阶段提供两个 shell 合同测试：

- 单二进制合同：证明最终只构建 `oz`，`cmd/wo` 消失，`oz flow` 入口真实可用，CI/Release 不再构建 `wo`。
- 无旧产品面合同：扫描活跃源码、文档、规格和测试，阻断 `wo` 命令名、`wo.yaml`、`cmd/wo`、`XDG_STATE_HOME/wo` 等旧合同残留。

执行阶段还必须跑 `go test ./...`，证明业务逻辑迁移后 Go 回归通过。

## 风险

这是明确的破坏性变更。已有个人机器上的 `wo` 二进制、`wo.yaml` 和旧状态目录不会自动迁移。因为用户明确说明目前只有个人使用，不考虑兼容性，所以该风险接受。

最大实现风险是批量替换误伤历史归档或业务字符串。执行阶段应先用结构化测试锁住当前产品面，再用 `ruplacer` 批量改名，最后人工审查 `internal/app` 的状态路径、配置路径和命令提示。
