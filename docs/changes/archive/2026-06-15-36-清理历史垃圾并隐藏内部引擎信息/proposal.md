# 清理历史垃圾并隐藏内部引擎信息

## 为什么需要做

`oz flow` 已经是当前产品入口，旧 `wo` 命令、旧配置、旧状态目录和 Dagu 运行时不应继续出现在活跃维护面。仓库里仍有大量 dated 根测试脚本直接引用 `cmd/wo`、`wo.yaml`、`.wo` 和 `$XDG_STATE_HOME/wo`，它们会让维护者误以为旧产品面仍需要维护。

同时，`go-dag` 是内部调度实现，不是用户需要选择、配置或理解的引擎。只要用户在帮助、配置、状态、graph、错误信息或文档里看到 `go-dag`，就会把内部实现误解成公开产品合同。

## 做什么

- 删除或迁移根目录 `docs/changes/archive/2026-06-15-36-清理历史垃圾并隐藏内部引擎信息/tests/2026-*` 历史 shell 测试，保留当前业务价值但改写为 `oz flow` 合同。
- 扩大活跃维护面扫描范围，覆盖根测试层和 `docs/changes/archive/2026-06-15-36-清理历史垃圾并隐藏内部引擎信息/tests/specs` 里的旧变量、旧路径、旧命令残留。
- 从用户文档、规格、默认配置、graph/status/help/error 输出中移除 `go-dag` 和 engine 选择叙述。
- 清理活跃源码中的旧 `wo` 命名和旧 `.wo` 路径兼容逻辑，除非它只存在于明确的旧输入拒绝测试中。
- 清理 Dagu、legacy-agent、opencode 等迁移残留，避免它们继续成为当前维护合同。

## 不做什么

- 不重写工作流调度实现。
- 不改变 `oz flow run/status/watch/restart/clean/config` 的业务语义。
- 不扫描或改写 `docs/changes/archive/**`，除非执行阶段发现活跃合同引用了归档内容。
- 不为了通过扫描而删除仍有当前业务价值的测试；这些测试应迁移命名和输入形态。

## 验收方式

验收以三个创建阶段契约测试为准：

- 用户可见面不得出现内部 engine 名称。
- 根历史测试层必须清理。
- 活跃维护面不得残留旧 `wo`、Dagu、legacy-agent/opencode 产品合同。

这些测试必须和 `go test ./...` wrapper 一起通过，确保清理没有破坏当前 Go 回归。
